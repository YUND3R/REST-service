Топология: два кластера (edge / gpu) как в deploy/k8s/clusters/README.txt, плюс основной класс GPU a100 и резерв rtx5000.

Кластер A — CPU (gateway, auth):
  kubectl label node <cpu-node> edu-ml.io/tier=edge --overwrite

Кластер B — основные GPU-ноды (A100):
  kubectl label node <gpu-main-a100> edu-ml.io/tier=gpu --overwrite
  kubectl label node <gpu-main-a100> edu-ml.io/gpu-class=a100 --overwrite

Резервная GPU-нода (RTX 5000 / A5000), куда планировщик уведёт pod при сбое/нехватке A100:
  kubectl label node <gpu-backup-rtx5000> edu-ml.io/tier=gpu --overwrite
  kubectl label node <gpu-backup-rtx5000> edu-ml.io/gpu-class=rtx5000 --overwrite

Taint на всех GPU-нодах (как обычно):
  kubectl taint nodes <gpu-node> nvidia.com/gpu:NoSchedule --overwrite

Поды worker-analyze, worker-generate и worker-pipeline могут работать только на нодах с gpu-class a100 или rtx5000;
предпочтение при планировании — a100.

ML HTTP-сервисы и ml-balancer не используются в минимальной worker-архитектуре.

Применение (сначала кластер B, затем A):
  kubectl kustomize --load-restrictor=LoadRestrictionsNone deploy/k8s/overlays/gpu-pool-rtx5000-a100/ml | kubectl apply -f -
  kubectl kustomize --load-restrictor=LoadRestrictionsNone deploy/k8s/overlays/gpu-pool-rtx5000-a100/edge | kubectl apply -f -

Чистый split без разделения классов GPU:
  см. overlays/cpu-gpu-topology/NODE_LABELS.txt
